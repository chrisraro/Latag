# Latag MVP — Design Spec

**Date:** 2026-07-13
**Status:** Approved pending final user review
**Source:** `Latag-App-Context.md` + product blueprint + brainstorming session decisions

---

## 1. Product Definition

Latag is an **offline-first pocket ops tool for ukay-ukay / vintage clothing resellers in the Philippines** (Naga City, Baguio, Metro Manila). It lets a curator log a garment in under 5 seconds, one-handed, in a no-signal warehouse; watch profit / capital-recovery math update live from the local database; and export listings to Instagram with one copy-to-clipboard.

**Primary user:** an independent reseller sourcing from warehouses/basements for up to 6-hour sessions, selling via Instagram drops.

**Core promise:** 100% offline, zero latency, zero typing in the item flow (the only text inputs in the app are session name/location and brand search).

## 2. Decisions Made (brainstorming outcomes)

| Decision | Choice |
|---|---|
| MVP scope | Core offline loop only — **no** Supabase auth, **no** telemetry backend in this phase |
| Deliverable order | Design system + HTML mockups first (impeccable skill), then the real Expo app |
| Sold flow | Record **actual** `soldPrice` + `soldAt`; dashboards show projected vs realized |
| Item types | **Tops only** — PTP + length measurements as specced; bottoms are v2 |
| Condition scale | Numeric chips: `10/10 · 9/10 · 8/10 · 7/10` |
| Ingestion UX | **One-screen Rapid Console** with sticky values between saves |

**Out of scope for MVP (deliberate):** auth/licensing, telemetry, bottoms/non-top categories, cross-session inventory search, backups/CSV export, editable condition scales, light theme.

**Added by the platform & monetization spec (companion doc):** the MVP additionally ships inside a pnpm **monorepo** (`apps/mobile` + stub `apps/web`) and includes the **free-tier log counter** — 20 lifetime item creations tracked in a local `entitlements` table, a remaining-logs indicator in the Rapid Console header once ≤ 10 remain, and a **Go Pro sheet** at 0 that links to the website (no in-app purchase flow). See `2026-07-13-latag-platform-monetization-design.md`.

## 3. Tech Stack (fixed by blueprint)

- **Mobile:** React Native / Expo (latest SDK), **Expo Router**, TypeScript strict.
- **Local DB:** `expo-sqlite` + `drizzle-orm/expo-sqlite`. Migrations generated by `drizzle-kit`, applied on app launch via Drizzle's expo-sqlite migrator.
- **Reactivity:** Drizzle `useLiveQuery` is the state manager. No Redux/Zustand.
- **Media:** `expo-camera` (capture), `expo-image-manipulator` (compress), `expo-file-system` (storage). URIs only in DB — never BLOBs.
- **Styling:** NativeWind (Tailwind for RN). **Animations:** React Native Reanimated. **Haptics:** `expo-haptics`. **IDs:** `expo-crypto` `randomUUID()`. **Clipboard:** `expo-clipboard`.

### Hard architectural rules (from blueprint — non-negotiable)

1. App functions 100% offline. Zero cloud reads for inventory/sessions/photos, ever.
2. Never store image BLOBs in SQLite; never upload images. Compress → local file → store `file://` URI string.
3. Media protocol: `expo-image-manipulator` (resize width 1200, compress 0.7, JPEG) → move to `${FileSystem.documentDirectory}latag_media/` → return `file://` URI.
4. OLED pure-black default theme.
5. Zero-typing ingestion: scroll wheels / sliders for dimensions, chips for category/condition. No text inputs for numbers.

## 4. Screen Map (Expo Router)

```
/                       Sessions List (home)
/session/new            New Session sheet (modal)
/session/[id]           Session Dashboard
/session/[id]/add       Rapid Console (item ingestion)
/session/[id]/export    IG Drop export
/item/[id]              Item Detail
/item/[id]/sold         Mark Sold sheet (modal)
```

### 4.1 Sessions List `/`
Cards per sourcing run: name, location, type badge (Selector / Bulto), item count, headline number (Selector → projected profit; Bulto → recovery %). Big `+ New Session` button. Newest first.

### 4.2 New Session `/session/new` (modal)
Fields: name (text), location (text, optional), type toggle (Selector | Bulto). Choosing Bulto reveals a bale-cost scroll wheel (₱500 steps). Create → navigates into the new dashboard.

### 4.3 Session Dashboard `/session/[id]`
Mode-adaptive header:
- **Selector:** Projected profit `SUM(targetSellPrice) − SUM(individualCost)` and Realized profit `SUM(soldPrice of sold) − SUM(individualCost of sold)`.
- **Bulto:** capital-recovery progress bar. **Realized recovery** = `SUM(soldPrice of sold) / totalBaleCost × 100`, filled solid. **Projected recovery** = `SUM(targetSellPrice of all items) / totalBaleCost × 100`, shown as a ghost marker on the same bar.

Below: item grid (photo thumbnail or placeholder, brand, price, sold badge), filter chips `All / Available / Sold`, long-press → multi-select mode → `Export to IG`. Prominent `+ Add Item` → Rapid Console.

### 4.4 Rapid Console `/session/[id]/add` — the core interaction
One screen, one-handed, target < 5 seconds per item after photos:

- **Header:** session name, running item count, running projected total (live).
- **Photo slots:** four tappable slots — Front / Back / Tag / Flaw — each opens camera inline, shows thumbnail once captured. Photos optional (never block save).
- **Brand:** horizontal chips of recent brands (derived: most-recent-distinct from this device's items) + `＋ search` chip opening a filtered type-ahead (the only text input in the item flow).
- **Category chips (tops only):** `Tee · Polo · Longsleeve · Jacket · Hoodie · Sweater · Jersey · Crewneck`.
- **Condition chips:** `10/10 · 9/10 · 8/10 · 7/10`.
- **Scroll wheels** (horizontal, haptic tick per detent): PTP inches (14–36, 0.5 steps), Length inches (20–36, 0.5 steps), Price ₱ (₱50–₱5,000; ₱10 steps below ₱500, ₱50 steps above). In **Selector** sessions an additional Cost ₱ wheel appears; in **Bulto** it is hidden and `individualCost` stays 0.
- **SAVE + NEXT:** one big button → haptic success + Reanimated flash → header count increments.
- **Sticky values:** brand, category, condition, and wheel positions persist to the next item; photo slots clear. Batches of similar items cost one tap each.
- Editing an existing item reopens this console pre-filled (route param).

### 4.5 Item Detail `/item/[id]`
Photo carousel with type labels (front/back/tag/flaw), all fields, status. Actions: `Mark Sold` (→ 4.6), `Edit` (→ Rapid Console pre-filled), `Delete` (confirm dialog; deletes photo rows **and** files).

### 4.6 Mark Sold `/item/[id]/sold` (modal)
Price wheel defaulting to `targetSellPrice` (haggling adjustment in one thumb swipe) → confirm → sets `status='sold'`, `soldPrice`, `soldAt`. Un-marking sold (mistake recovery) resets the three fields from Item Detail.

### 4.7 IG Export `/session/[id]/export`
Item picker defaulting to all **Available** items (toggle any), live caption preview, `Copy to Clipboard` (expo-clipboard) + success haptic/toast. Export is read-only — it never mutates items. Exact caption format per item:

```
👕 [Brand] [Category]
📏 Size: (PTP: [ptpInches]" | L: [lengthInches]")
✨ Condition: [condition]
💸 ₱[targetSellPrice]
📍 Comment "Mine" to claim
---
```

## 5. Data Layer

### 5.1 Schema (Drizzle / expo-sqlite)

Blueprint schema **verbatim**, with three additions on `items` (justified deviations):

```ts
import { sqliteTable, text, real, integer } from 'drizzle-orm/sqlite-core';

export const sessions = sqliteTable('sessions', {
  id: text('id').primaryKey(),
  name: text('name').notNull(),
  type: text('type', { enum: ['selector', 'bulto'] }).notNull(),
  totalBaleCost: real('total_bale_cost').default(0),
  location: text('location'),
  createdAt: integer('created_at', { mode: 'timestamp' }).notNull(),
});

export const items = sqliteTable('items', {
  id: text('id').primaryKey(),
  sessionId: text('session_id').references(() => sessions.id).notNull(),
  brand: text('brand').notNull(),
  category: text('category').notNull(),
  ptpInches: real('ptp_inches').notNull(),
  lengthInches: real('length_inches').notNull(),
  condition: text('condition').notNull(),
  individualCost: real('individual_cost').default(0),
  targetSellPrice: real('target_sell_price').notNull(),
  status: text('status', { enum: ['available', 'sold'] }).default('available'),
  // ── additions ──
  soldPrice: real('sold_price'),                                   // actual price when sold (nullable)
  soldAt: integer('sold_at', { mode: 'timestamp' }),               // when sold (nullable)
  createdAt: integer('created_at', { mode: 'timestamp' }).notNull(), // ordering
});

export const photos = sqliteTable('photos', {
  id: text('id').primaryKey(),
  itemId: text('item_id').references(() => items.id).notNull(),
  localUri: text('local_uri').notNull(),
  type: text('type', { enum: ['front', 'back', 'tag', 'flaw'] }).notNull(),
});
```

### 5.2 Access pattern
- Reads: `useLiveQuery` per screen; UI never caches inventory in React state.
- Writes: thin repository layer `lib/repo/` (`createSession`, `addItem`, `markSold`, `unmarkSold`, `deleteItem`, …). Screens call repo functions, never raw Drizzle.
- Ephemeral-only React state: Rapid Console inputs + sticky values (small context).

### 5.3 Money math — `lib/math.ts`
Pure functions, no DB, no React. All money as `real` pesos; single `formatPeso()` display util (`₱1,250`).

- `selectorProjected(items) = Σ target − Σ cost`
- `selectorRealized(soldItems) = Σ soldPrice − Σ cost(sold)`
- `bultoRealizedRecoveryPct(soldItems, baleCost) = Σ soldPrice / baleCost × 100`
- `bultoProjectedRecoveryPct(items, baleCost) = Σ target / baleCost × 100`
- Edge cases specified: empty session → 0s; `baleCost = 0` → recovery renders `—` (never divide by zero); sold-below-cost → negative realized allowed and shown in red.

### 5.4 Media pipeline — `lib/media.ts`
Single capture function implementing the blueprint protocol exactly: camera capture → manipulator (width 1200, compress 0.7, JPEG) → `FileSystem` move to `documentDirectory/latag_media/` → return `file://` URI → caller inserts `photos` row.

**Write ordering (deliberate):** file is written before the DB row. A crash between the two leaves an orphan **file**, never a DB row pointing at nothing. An **orphan sweep** on app start deletes files in `latag_media/` with no matching `photos.localUri`. Deleting an item deletes its photo rows and files.

## 6. Error Handling

| Failure | Behavior |
|---|---|
| DB write fails | Console state preserved (nothing cleared), error haptic + toast. No silent loss. |
| Camera permission denied | Photo slots show "grant access" state; saving items still works (photos never required). |
| Photo file missing at render | Placeholder tile; DB row kept. |
| Bale cost 0 in Bulto math | Render `—`, never divide by zero. |
| Delete item | Confirm dialog → transactional: photo files + rows + item row. |

## 7. Design System — "Warehouse Console"

Pro-equipment feel, not social-app feel. Full token sheet written as `docs/design-system.md` during the mockup phase (impeccable skill) and locked by mockup approval.

- **Surfaces:** pure `#000000` base (OLED); elevated `#111111` / `#1A1A1A`; hairline borders, no shadows.
- **Accent:** single acid green ≈ `#B8F135` — money-positive numbers, progress, SAVE. Red-orange strictly for destructive/error. Sold badge = neutral gray.
- **Type:** system sans; tabular numerals for all money/measurements (no digit jitter on wheels); peso figures 1.5–2× body size — money is the hero.
- **Touch:** 4pt spacing grid; targets ≥ 48px; large pill chips; haptic tick per wheel detent; dark-only.

## 8. Testing

- **Unit (highest value):** every `lib/math.ts` formula incl. edge cases; IG caption formatter (exact string match to template).
- **Integration:** Drizzle schema round-trips on in-memory SQLite; repo flows (create session → add item → mark sold → totals correct); orphan-sweep behavior.
- **Manual device QA checklist:** camera → compress → URI pipeline; haptics; wheel feel; full airplane-mode session end-to-end. Hardware paths get a checklist, not mocks.

## 10. Phase Plan

1. **Spec** — this document. ✅
2. **Design mockups** — design system + phone-framed HTML mockups via **impeccable**; published as a clickable artifact. Gate: user approves the look. *Expanded during this phase to the full production surface (15 screens): the MVP core loop (sessions, dashboards, Rapid Console, camera, item detail, mark-sold, IG export, empty states) plus first-run screens (welcome/activation, email OTP, onboarding ×2) and Settings, designed ahead of their build phases. 4. **Parked for later phases:** web platform (landing, portal, admin console, payments, legal pages — see companion platform spec), Next.js + Upstash telemetry, Supabase auth/licensing + license receipt caching, bottoms & other categories, CSV export/backup.

## Appendix: Deliberate Implementation Deviations (Phase A build)

Recorded at whole-branch review so these stay decisions, not surprises:

1. Dashboard multi-select for export replaced by an always-visible **Export chip** (long-press multi-select deferred).
2. Bulto **projected recovery** renders as a text line under the bar, not a ghost marker on the bar (marker deferred to polish).
3. Brand entry is recent-brand chips + a plain text field (filtered type-ahead list deferred).
4. The Reanimated acid **save flash** is deferred to the polish pass (save feedback ships as haptic success + counter update).
